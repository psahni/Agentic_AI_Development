# CFO Agent — Full Documentation

## What Is This?

The CFO Agent is a **multi-agent AI system** that automatically analyses a company's
financial data, detects risks and anomalies, pauses for a human to review the findings,
and then writes a boardroom-ready CFO report — all without manual intervention beyond
one approval click.

It is built on **LangGraph**, a framework for building stateful, graph-based AI
pipelines where multiple specialised agents collaborate on a shared task.

---

## Libraries Chosen

| Library | Role | Why |
|---|---|---|
| **LangGraph** | Graph orchestration engine | Manages the flow between agents, handles state, and provides SQLite memory out of the box |
| **LangChain Core** | Message types & LLM abstractions | Provides `SystemMessage`, `HumanMessage`, and `RunnableConfig` — the standard interface for talking to LLMs |
| **langchain-groq** | Groq LLM client | Connects to Groq's API for fast, cheap inference via Llama models |
| **Groq (llama-3.3-70b / llama-3.1-8b)** | The AI brains | Two different models — one smart for routing decisions, one fast for writing tasks |
| **pandas** | Data manipulation | Industry-standard for tabular data; reads CSV, calculates metrics, filters anomalies |
| **SQLite (via SqliteSaver)** | Persistent memory | Saves the full graph state after every node so runs can be resumed if interrupted |
| **python-dotenv** | Environment config | Loads `GROQ_API_KEY` from `.env` so secrets never touch the code |
| **pathlib** | File paths | Cross-platform absolute paths — reports and database always save to the right folder regardless of where you launch the script from |

## How It Works — The Mental Model

Think of this system like a **project manager (Supervisor) directing a team of
specialists (Workers)**:

- The Supervisor never does the actual work. It only reads the current situation
  and decides who should act next.
- Each Worker does one focused job, updates the shared state, and hands control
  back to the Supervisor.
- This loop continues until the report is written or an error stops the pipeline.

The shared state is like a **backpack** that gets passed between every agent.
Each agent reads from it and writes their results back into it.

---

## The State — Shared Backpack

Defined in [state.py](state.py). Every field is visible to every agent.

```
CFOState
│
├── thread_id       → session identifier for SQLite ("cfo-2024-annual")
│
├── raw_data        → the CSV loaded as a JSON string
├── metrics         → dict of all calculated financial numbers
├── anomalies       → list of risk strings, accumulates across runs
├── insights        → LLM narrative interpretation of the numbers
├── report          → the final written CFO markdown report
│
├── next_worker     → supervisor writes the next agent name here
├── hitl_approved   → True once the human approves at checkpoint
├── status          → "ok" or "error"
└── error_message   → human-readable reason if status is "error"
```

`anomalies` uses `Annotated[list, operator.add]` — this means LangGraph
**appends** new anomalies to the existing list instead of replacing it,
so findings accumulate safely across retries.

```

│ Reads   │   │ Calcul- │   │ Rule │   │ Pauses for │   │ LLM    │
│ CSV →   │   │ ates    │   │ +LLM │   │ human      │   │ writes │

```

### Detailed Step Sequence

**Step 1 — START → Supervisor**
`main.py` calls `app.invoke(initial_state, config)`. LangGraph immediately
routes to the Supervisor node.

**Step 2 — Supervisor → Loader**
The Supervisor reads state. Everything is empty. Rule 1 says: "loader must
always run first." Supervisor writes `next_worker = "loader"` and returns.
The conditional edge routes to the Loader.

**Step 3 — Loader runs**
Reads `data/financials.csv`, validates all required columns exist, converts
the DataFrame to a JSON string, and writes it to `state["raw_data"]`.
Returns to Supervisor.

**Step 4 — Supervisor → Analyst**
Supervisor sees `raw_data` is populated, `metrics` is empty. Rule 2: "analyst
runs after loader." Routes to Analyst.

**Step 5 — Analyst runs**
Reads the JSON back into a DataFrame. Calculates: annual revenue, profit,
margins, revenue growth, cash runway, burn rate, budget variance, and a
full month-by-month breakdown. Writes everything to `state["metrics"]`.
Returns to Supervisor. No LLM used — pure pandas arithmetic.

**Step 6 — Supervisor → Risk**
Supervisor sees metrics populated, anomalies list is empty. Rule 3: "risk
runs after analyst." Routes to Risk worker.

**Step 7 — Risk worker runs**
Scans every row in the DataFrame against 5 threshold rules:
- Revenue missed budget by more than 15%
- Net margin below 5%
- Cash balance below 2 months of expenses
- R&D spend more than 1.8× the monthly average
- Tax payment more than 2× the monthly average

Each breach is appended to `state["anomalies"]` as a human-readable string.
Returns to Supervisor.

**Step 8 — Supervisor → Checkpoint**
Risk is done (metrics are populated). Rule 4: "checkpoint always runs after
risk." Routes to Checkpoint.

**Step 9 — Checkpoint runs (Human-in-the-Loop)**
Prints a formatted summary of all findings to the terminal. Blocks and waits
for the human to type `yes` or `no`. If `yes`: sets `hitl_approved = True`
and returns. If `no`: sets `status = "error"` and returns.

**Step 10 — Supervisor → Writer or FINISH**
If human approved → Rule 5: "writer runs only after human_approved is YES."
Routes to Writer.
If human rejected → Rule 7: "if status is error, respond FINISH." Graph ends.

**Step 11 — Writer runs**
Makes two sequential LLM calls:
1. **Insights call** — send metrics + anomalies to the LLM, ask for 4 business
   insights in plain English
2. **Report call** — send metrics + anomalies + insights to the LLM, ask for a
   full structured markdown CFO report

Saves the report to `reports/cfo_report_YYYYMMDD_HHMMSS.md`.
Returns to Supervisor.

**Step 12 — Supervisor → FINISH**
Supervisor sees the report is written. Rule 6: "once report is written, respond
FINISH." The conditional edge maps `"FINISH"` to `END`. Graph terminates.

**Step 13 — main.py prints the report**
`app.invoke()` returns `final_state`. If report exists, it's printed to terminal.

## The Supervisor — Deep Dive

The Supervisor is the most important component. It is an **LLM-based router**.

```python
# Every call gives the LLM a snapshot of the pipeline:
Pipeline status:
- Data loaded:        YES
- Metrics calculated: YES
- Anomalies found:    4 detected
- Risk assessed:      YES
- Human approved:     NO
- Report written:     NO
- Current status:     ok

# And asks: "Which worker should run next?"
# The LLM must reply with exactly ONE word.
```

### Why use an LLM for routing?

Simple `if/else` routing is brittle. An LLM-based supervisor can handle nuance:
- "Skip checkpoint if there are no anomalies and margins are healthy"
- "Re-run risk if the analyst found unusually high variance"
- "Go to FINISH early if an error occurred mid-pipeline"

These require *reasoning*, not just conditionals.

### Safety guard

If the LLM returns anything unexpected (e.g. "I think loader should run"),
the supervisor defaults to `FINISH` rather than getting stuck in an infinite
loop. This is a critical production pattern.

```python
valid = ["loader", "analyst", "risk", "checkpoint", "writer", "finish"]
if next_worker not in valid:
    next_worker = "finish"   # safe fallback
```

### Two-model strategy

```
supervisor_llm → llama-3.3-70b-versatile   (smart, slower, more expensive)
worker_llm     → llama-3.1-8b-instant      (fast, cheap, focused)
```

The Supervisor makes the most consequential decision in the system — a wrong
routing choice breaks the entire pipeline. It uses the larger model.

Workers have focused, structured tasks (calculate this, write that) where the
smaller model is accurate enough and 5× cheaper.

## SQLite Memory — How State Persists

```
Every node that completes → SqliteSaver writes full state to memory.db
                                          ↓
                              keyed by thread_id + checkpoint_id
```

This means:

- If the program **crashes** mid-run, restarting with the same `thread_id`
  resumes from the last completed node — no work is repeated.
- Different `thread_id` values are completely isolated sessions. `"cfo-2024-annual"`
  and `"cfo-2024-q1"` have separate memories.
- You can inspect `memory.db` with any SQLite browser to see every saved state.

## Design Decisions Worth Knowing

**Why no LLM in Loader or Analyst?**
Those workers deal with facts — read a file, compute a number. Facts do not
need language intelligence. Using pandas is faster, cheaper, and more accurate
than asking an LLM to do arithmetic. LLMs enter only when *reasoning* or
*writing* is needed (Risk assessment + Writer).

**Why does every worker return to the Supervisor instead of chaining directly?**
Because the Supervisor can make context-aware decisions at each step. If the
Loader fails, the Supervisor routes to FINISH immediately — no need for every
worker to handle every failure case themselves. This keeps workers simple and
the routing logic centralised.

**Why store the DataFrame as a JSON string?**
LangGraph state must be serialisable — plain Python types only. A pandas
DataFrame is not. Converting to JSON string on write and back to DataFrame
on read is the standard pattern (package it for shipping, unpack at destination).

**Why two LLM calls in Writer instead of one?**
Separating "generate insights" from "write the report" means each call has a
single focused task. One combined call would ask the LLM to analyse AND write
simultaneously — lower quality on both. Chaining two focused calls beats one
unfocused call every time.
